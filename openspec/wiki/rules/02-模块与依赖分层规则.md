# 模块与依赖分层规则

## 文档说明

- 覆盖规则：前端、Python 后端和通用层的模块边界与依赖方向。
- 核心要求：依赖必须单向流动；公共层不能反向依赖业务层；router 不得直接依赖跨域实现细节。
- 应用场景：新项目初始化、模块重构、设计阶段划分目录边界。
- 命中主题：模块分层、依赖方向、边界划分
- 触发关键词：模块划分,依赖关系,layer,boundary,monorepo
- 关联 skill：common/init-agent、common/init-tech、python/backend-design
- 前置规则：00、01
- 后置规则：03、14、23
- 排除场景：单文件局部修复

## 推荐分层

### Python 后端

- `router`：HTTP 入口、参数接收、鉴权
- `service`：业务编排、事务、跨仓储协作
- `repository`：数据库与外部存储访问
- `schema`：请求响应模型、领域输入输出模型
- `domain`：纯业务对象、状态机、规则
- `common`：日志、配置、错误、工具

### 前端

- `pages`：页面容器
- `components`：可复用 UI 组件
- `hooks/store`：状态和副作用管理
- `api/service`：请求封装和数据层
- `shared`：跨模块公共能力

## 依赖规则

1. router -> service -> repository 是主方向。
2. `common` 不依赖具体业务模块。
3. repository 不回调页面语义或 UI 逻辑。
4. service 不依赖具体页面组件。
